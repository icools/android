# BDD 與 TDD：差異、搭配方式與我會怎麼用

很多人在學測試時，常常會把 `BDD` 跟 `TDD` 混在一起。

它們確實有關，但不是同一件事情。

## TDD 比較像什麼

`TDD` 是 `Test-Driven Development`。

它比較偏工程師的開發節奏，也就是：

- 先寫失敗的測試
- 再寫最少量的實作讓測試通過
- 再重構

它的重點不是「測試寫很多」，而是讓測試反過來約束設計。

這種方式的好處是，你比較不容易一開始就把東西寫太大坨，因為你會被測試逼著把責任切小、介面想清楚。

## BDD 比較像什麼

`BDD` 是 `Behavior-Driven Development`。

它更強調的是「行為」和「共同語言」。

也就是說，它關心的不是只有工程師怎麼寫，而是：

- 使用者期待什麼行為
- 業務規則是什麼
- 團隊能不能用一致的語言描述需求

所以 BDD 比起單純測試技巧，更像是需求描述與驗收思維。

## 一句話分辨

如果要很白話地分：

- `TDD`：比較像工程實作的節奏控制器
- `BDD`：比較像需求與行為的對齊工具

## 我會怎麼搭配它們

我自己覺得最務實的方式不是選邊站，而是把兩者放在不同層。

### 在需求層，用 BDD 思考

先把這個功能對外應該怎麼表現講清楚。

例如：

- 使用者按下送出時，如果欄位缺漏，應該顯示錯誤
- 如果 OTP 驗證成功，應該跳到下一頁
- 如果 API 回傳逾時，應顯示可重試的狀態

這一層是在定義行為，而不是急著寫 implementation。

### 在實作層，用 TDD 控制品質

當行為被說清楚之後，再往下拆成：

- ViewModel 的判斷
- use case 的輸入輸出
- repository 的行為
- formatter 或 mapper 的規則

這時候就很適合用 TDD，讓邏輯在小單位內可以被穩定驗證。

## 哪些地方很適合 TDD

我覺得下面這些地方很適合：

- 純邏輯轉換
- 狀態切換規則
- 驗證規則
- domain use case
- 資料 mapping
- 邊界條件容易很多的模組

## 哪些地方比較適合先從 BDD 下手

下面這些情境，直接先談行為會更有幫助：

- 產品需求容易反覆變動
- 很多角色都要參與討論
- 驗收標準常常講不清楚
- UI 流程和商業規則綁很緊

因為你如果一開始就只談測試實作，很容易寫得很勤勞，但最後測的是錯的東西。

## 我不太喜歡的誤區

### 誤區 1：把 BDD 當成只是寫 Given / When / Then

那只是表達格式，不是本體。

真正重要的是，你有沒有把行為講清楚。

### 誤區 2：把 TDD 當成只是多寫測試

TDD 的核心其實是用測試引導設計，而不是讓 coverage 看起來很漂亮。

### 誤區 3：什麼都硬套

不是所有東西都值得用很重的方式處理。

有些 UI 細節、一次性頁面、純黏合層，硬套完整 TDD / BDD 反而會讓你邊寫邊翻白眼。

## 我會怎麼實際落地

如果是一般產品開發，我會比較偏這樣：

1. 先用 BDD 思維把功能行為講清楚
2. 把驗收條件列出來
3. 把核心邏輯拆小
4. 對核心邏輯採 TDD
5. 對整體流程補 UI / integration 測試

這樣不會太重，也比較貼近真實世界。

## 我的結論

我不會把 `BDD` 跟 `TDD` 看成互相競爭的流派。

它們比較像是在不同層級幫你減少誤解：

- `BDD` 幫你避免需求講錯
- `TDD` 幫你避免邏輯寫壞

兩個一起用，反而比較完整。
