# Paddy 的 Coding、Design Pattern 與 DDD 學習筆記

這篇先把我想補進知識庫的一條學習脈絡整理下來：

不是只學怎麼把 code 寫得能動，而是去學怎麼觀察行為、怎麼做建模、怎麼看待設計與程式的責任分配。

## 我會怎麼定位這條學習線

如果只講很白話的感覺，我會把這個方向理解成：

- 不只是教你語法
- 不只是教你套 pattern 名字
- 而是帶你去看程式背後的行為、結構與設計思維

這種內容很適合已經寫了一段時間程式，但開始覺得：

- 為什麼系統越寫越亂
- 為什麼功能明明做得出來，卻很難改
- 為什麼團隊討論常常講不到同一件事
- 為什麼 code review 常常只在修表面問題

如果你開始碰到這些感覺，那這種設計導向的學習就很值得投入。

## 我為什麼會想推薦

因為真正讓工程師拉開差距的，往往不是「會不會寫功能」，而是：

- 你有沒有能力把問題講清楚
- 你能不能把責任切乾淨
- 你能不能看出壞味道
- 你能不能在複雜需求裡維持可讀性

這些能力，不太可能只靠背 API 長出來。

## 我會期待學到的東西

### 1. Design Pattern 不是拿來背招式名

而是幫你辨識常見結構問題。

### 2. DDD 不是拿來堆術語

而是幫你把真實世界的問題、語言與邊界拉進程式。

### 3. 行為比 class 更重要

很多系統之所以難維護，不是因為 class 太少，而是因為行為混在一起。

### 4. 會建模，比會堆功能更有價值

因為一旦模型錯了，後面再怎麼修都只是把房子蓋得更歪。

## 我覺得特別適合誰

我覺得這條學習線特別適合：

- 已經工作一段時間的工程師
- 開始接觸架構與模組設計的人
- 想把 code review 做深一點的人
- 想理解 DDD、Pattern 與設計思維怎麼連起來的人

## 我的簡單結論

如果只是想快速把功能做出來，這類內容一開始可能會覺得有點慢。

但如果你希望自己未來不是只會一直堆功能，而是能真的看出問題、講出問題、拆出問題，那這種設計與行為導向的內容，我覺得很值得長期跟。
