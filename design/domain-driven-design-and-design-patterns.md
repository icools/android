# DDD 與 Design Pattern：差別、關係與我會怎麼看

很多人在學軟體設計時，會先接觸 `Design Pattern`，之後又聽到 `DDD`，然後腦袋裡開始冒出一個問號：

這兩個到底差在哪裡？

## 先講最白話的差別

如果要一句話講：

- `Design Pattern` 比較像常見設計問題的解法模板
- `DDD` 比較像理解業務領域、建立模型與切系統邊界的方法

所以它們不是同一層的東西。

## Design Pattern 比較在解什麼

`Design Pattern` 通常在處理的是比較常見、可重複出現的設計問題。

例如：

- 物件建立方式
- 角色協作方式
- 狀態怎麼切
- 行為怎麼替換
- 依賴怎麼隔離

這一類模式很像是工程世界的慣用招式。

它能幫你提升可讀性、降低重複造輪子，也讓團隊比較容易有共同理解。

## DDD 比較在解什麼

`DDD` 全名是 `Domain-Driven Design`。

它關心的重點不是某個 class 該不該用 factory，而是：

- 這個系統真正要處理的領域問題是什麼
- 團隊講的語言和程式裡的語言有沒有一致
- 哪些概念才是核心模型
- 邊界要怎麼切
- 哪些上下文其實不該混在一起

也就是說，DDD 比較偏高階的建模思維。

## 它們的關係像什麼

我會把它們想成這樣：

- DDD 幫你決定「你到底在處理什麼世界」
- Design Pattern 幫你處理「這個世界裡的程式結構要怎麼寫得漂亮一點」

前者偏問題建模。

後者偏結構手法。

## 為什麼很多人會混在一起

因為兩者最後都會落到 code 上。

但它們的起點不同。

一個是先理解問題域。

一個是先面對結構上的常見設計問題。

如果你先把模式背很多，但沒有搞清楚 domain，最後很可能會寫出一堆很工整、很有型，但跟真實需求其實沒有那麼貼合的東西。

## 我覺得 DDD 最有價值的地方

不是那些名詞很帥，而是它逼你去問：

- 我們到底在解什麼問題
- 這些詞是誰定義的
- 這些概念應不應該放在同一個模組
- 我們程式裡的語言，和使用者或業務的語言是不是對得上

只要這幾個問題開始被認真問，系統通常就比較不容易變成巨大垃圾堆。

## 我覺得 Design Pattern 最有價值的地方

它讓你少繞很多冤枉路。

例如你看到某種結構需求時，腦中會知道：

- 這像 state pattern
- 這像 strategy
- 這適合 observer
- 這該考慮 factory 或 builder

這些模式不是要你炫技，而是讓你快速辨識問題結構。

## 很實際的使用方式

如果今天在做一般產品開發，我會比較傾向這樣：

1. 先用 DDD 思維理解領域與邊界
2. 再決定哪些模型、流程與責任要怎麼切
3. 實作時，再用 Design Pattern 補足局部設計問題

也就是說，我不會反過來從模式開始想整個系統。

不然很容易變成：

「我今天學了某個 pattern，所以整個世界看起來都像釘子。」

## 我的結論

如果你把 `DDD` 和 `Design Pattern` 看成競爭關係，就有點搞錯了。

它們比較像上下層互補：

- `DDD` 幫你理解問題空間
- `Design Pattern` 幫你優化解法空間

一個比較像地圖。

一個比較像工具箱。

兩者都重要，但順序不要顛倒。
