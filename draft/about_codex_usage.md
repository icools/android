

# 我開始把 Codex 當成工程工作台，而不只是寫程式工具

一開始我對 Codex 的想像其實很單純。

我原本以為，它大概就是一個比較強的 AI coding assistant：可以幫我補 code、改 bug、整理一小段邏輯，或是幫我快速理解某個 class 在做什麼。

但實際用一段時間之後，我慢慢發現，Codex 真正有趣的地方，可能根本不是「幫你寫幾行程式」而已。

它更像是一個可以承接工程工作流的操作台。

不是只處理單一檔案，不是只回答單一問題，而是開始有機會參與整個開發過程：理解專案、拆解規格、整理重構方向、讀取多個 project 的脈絡、協助驗證、補文件，甚至串接外部工具與知識來源。

這種感覺，和我一開始想像的「純寫 code 工具」真的差很多。

## 我後來發現，Codex 比較像什麼？

如果要用一句比較白話的方式來說，我現在會把它看成：

> 一個可以跨 repo、可吃規格、可產生 diff、可串接工具、也可以協助長流程任務的工程代理工作台。

這個定位一改，整個使用方式就完全不一樣了。

以前你可能只是丟一句：

- 幫我修這個 bug
- 幫我補這段 code
- 幫我重構一下這個 function

但現在比較有價值的用法，反而是這種：

- 幫我先整理這個問題的 lifecycle 時序
- 幫我分析目前責任分界是不是太繞
- 幫我根據 diff 產出工作日誌與重構說明
- 幫我把 A project 的概念整理後寫進 B project 文件
- 幫我先讀 specs、plans，再按順序實作與驗證

當你開始這樣用它，你會感覺它的角色不再只是 code generator，而是比較像一個會幫你做工程整理、協助執行、補足文件與驗證流程的副手。

## 對我來說，最值得注意的幾個功能

### 1. 不只是單次對話，而是可以平行處理多個工程任務

這一點很重要。

因為真實工作不會只有一條線。

很多時候我們會同時卡在幾件事上：

- 一邊查 Android bug
- 一邊整理重構方向
- 一邊補文件
- 一邊思考某個 side project 要怎麼延伸

如果 Codex 只是單輪問答，它的價值其實有限。

但當它可以被拿來分工處理不同任務時，感覺就完全不同了。

你可以想像成：

- 一個 agent 幫你看 root cause
- 一個 agent 幫你整理規格與改法
- 一個 agent 幫你寫 README 或工作日誌
- 一個 agent 幫你做驗證與收尾整理

這種做法的價值，不只是省時間，而是減少你腦中一直切換上下文的負擔。

### 2. AGENTS.md 這種機制，真的很適合有固定風格的人

如果你本身對程式風格、命名、架構邊界有明確要求，那 AGENTS.md 這種東西非常有用。

因為很多人和 AI 合作時，最痛苦的一件事就是：每次都要重新講一遍。

像是：

- 命名要清楚
- 優先 early return
- 不要亂加註解
- UI 和 Route/Screen 職責要分清楚
- 涉及 lifecycle、service、flow 時先整理事件順序
- 先給結論，再講風險，再給建議

這些如果每次都要重新提醒，其實很累。

但如果能寫進固定規則，AI 就比較像是在你的工作框架裡做事，而不是每次都用自己的方式重新發散。

我自己很喜歡這種感覺。

因為它不是在取代你的方法，而是在吸收你的方法。

### 3. Skill 的概念，比「提示詞魔法」更實際

我後來越來越覺得，很多人會把 AI 用得很不穩，原因不是模型不夠強，而是流程沒有被固定下來。

如果今天只是靠一次一次的 prompt 去堆，當然很容易飄。

但如果你把常見工作做成固定 skill，整體就穩很多。

例如：

- 讀 log 後整理 root cause 假設
- 根據 commit diff 產出主管版工作摘要
- 把討論內容轉成 spec
- 檢查 Android Auto 相關流程中的 race condition
- 先分析 class 責任，再給重構提案

這些事情一旦被固定成可重複的流程，Codex 的價值就會往上跳一個層級。

它就不只是「幫你答題」，而是變成一個會遵守套路、可複用的工作模組。

### 4. 多 project 參考能力，真的很適合做文件與知識轉移

這也是我覺得很實用的一點。

因為真實開發很常不是只看一個 repo。

有時候你需要：

- 參考舊專案的設計
- 從另一個 side project 抽出可重複利用的概念
- 把 A project 的介紹整理進 B project
- 綜合多個 project 的做法，產出一份比較整理文件

這種需求，如果全部手動做，常常很花時間。

但如果你已經有一套清楚的資料結構，或者先把各 project 的重點整理成 summary，再讓 Codex 幫你整合，整體效率會高很多。

我自己現在比較傾向的做法是：

不要期待它自動通靈全部 repo，而是先替每個 project 準備比較乾淨的摘要文件，讓它在比較穩定的 context 下去寫整理、比較與介紹。

這樣品質通常比直接硬掃整包原始碼更好。

### 5. 長流程任務，比單點改 code 更有價值

我越來越覺得，單點改 code 其實只是 AI 的基本功。

真正更有意思的是，當它可以參與一個完整流程：

1. 先讀規格
2. 再看現況
3. 列出計畫
4. 進行修改
5. 驗證結果
6. 補上文件

這種模式才比較接近現實工程。

也是從這個角度看，我會認為 Codex 最有價值的地方，不是讓你少打一點字，而是讓你開始可以把一些工程節奏標準化。

## 我自己的心得：它最迷人的地方，不是自動化，而是可承接性

很多工具都會強調自己能自動化很多事情。

但我現在覺得，Codex 最有意思的地方不是「它能不能自己做很多」，而是「它能不能接住你原本的工作方式」。

這件事很關鍵。

因為工程工作很少真的是完全從零開始亂做一通。

大多數時候，我們其實都有自己的習慣：

- 先分析現況
- 先畫責任邊界
- 再決定要不要改
- 修改後要有驗證
- 最後要能留下文件、摘要或工作紀錄

如果一個 AI 工具只能回答問題，卻接不住這種流程，它的價值最後就容易停在「偶爾好用」。

但如果它能慢慢融入這種流程裡，情況就不同了。

它會開始像一個熟悉你做事方式的工程副手。

不是每次都重新學一次，而是越用越能對齊你的節奏。

## 我目前認為最實用的三個方向

### 第一個：把個人偏好與工作守則固定下來

像是：

- 回覆格式
- 命名原則
- 架構偏好
- 分析順序
- 驗證習慣

這種東西越早固定，後面越輕鬆。

### 第二個：把常做的工程任務整理成模板或 skill

比起每次重新下 prompt，我更相信固定流程的價值。

例如：

- bug triage 模板
- spec 轉文件模板
- diff 轉工作日誌模板
- 多專案比較整理模板

這些東西一旦建立起來，後面會越來越順。

### 第三個：不要只拿它來寫 code，要拿它來整理脈絡

我覺得這是很多人可能還沒真正用到的地方。

寫 code 只是表面。

真正耗腦的是：

- 理解脈絡
- 整理問題
- 找出責任邊界
- 把思考留下來
- 讓之後的人也能看懂

如果 Codex 可以幫你承接這些部分，那它的價值就不只是開發效率，而是整體工程可理解性都會變好。

## 結語

我一開始真的以為，Codex 只是一個把工具集中起來的 UT 專案。

但實際試過之後，我覺得它更像是一個可以長成你自己工作流骨架的東西。

如果你只是拿它來手動點幾個按鈕，可能會覺得還好。

可是當你開始思考：

- 能不能讓 AI 接手操作
- 能不能把 app 內部狀態一起接上來
- 能不能讓它不只是會回答，而是真的理解我的工程習慣

它的價值就會開始往上拉。

至少對我來說，這就是我目前覺得它最值得玩的地方。
